# v2.0.3 Seed And Config Inventory

Status: active  
Updated: 2026-03-15

## Purpose

Inventory the currently visible checked-in config, defaults, examples, vault,
and memory-backed seed surfaces across the active family so `v2.0.3` Round B
can align them without guessing.

## Reading Rule

Treat this as an inventory and placement guide, not as a replacement for the
family config/state contract.

Canonical contract reference:

- `uDOS-dev/docs/v2.0.3-family-config-state-alignment.md`

## Surface Classes

Use these meanings consistently:

- `config/`: checked-in config examples, templates, and non-secret defaults
- `defaults/`: checked-in transitional defaults for shared workspace or starter
  behavior
- `examples/`: runnable or readable sample payloads and sessions
- `vault/`: tracked Markdown-first sample or teaching data
- `memory/`: mutable local runtime state, never canonical tracked seed truth

## Current Strong Patterns

The family already shows a few stable patterns worth preserving:

- `uDOS-core` owns environment-root vocabulary and contract shapes
- `uHOME-server` separates checked-in defaults, tracked vault samples, and
  mutable runtime config clearly
- `uDOS-empire`, `uHOME-client`, `uHOME-matter`, `uDOS-plugin-index`, and
  `uDOS-themes` mostly use `examples/` as their starter seed lane
- `sonic-screwdriver` clearly separates tracked vault examples from mutable
  runtime memory

## Current Drift Or Gaps

- not every repo uses the same language for “seed”, “defaults”, and “example”
- some repos still have placeholder `config/README.md` lanes without concrete
  examples
- there is not yet one family-level rule for when a starter payload belongs in
  `defaults/` versus `examples/`
- `memory/` usage is clear in some repos but undocumented in others

## Placement Rule For v2.0.3

Until a stricter contract is promoted, use this family rule:

- put shared variable names and root rules in `uDOS-core`
- put non-secret checked-in config examples in `config/`
- put starter payloads and operator walkthrough samples in `examples/`
- put tracked Markdown-first teaching or sample domain records in `vault/`
- use `defaults/` only when a repo truly has a reusable checked-in default
  profile that is not just an example
- never treat `memory/` as tracked seed truth
