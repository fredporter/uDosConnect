## Language & Runtime Stack

**Source:** `@dev/inbox/decisions/uCODE-v2-LANGUAGE_RUNTIME_SPEC.md`  
**Owner:** uDOS-core  
**Status:** locked for v2.1

### Architecture

| Layer | Language | Purpose |
|-------|----------|---------|
| Core runtime | Go | Source of truth for deterministic execution |
| Command language | uCode | Deterministic command DSL and workflow syntax |
| UI runtime | TypeScript | Presentation, components, grid, visualization |
| Script artifacts | Markdown (`.md`) | Executable scripts with uCode blocks |
| Helper scripting | Python (optional) | Data transforms, utilities (no runtime semantics) |

### Key Principle

**The language stack preserves a single Runtime Truth.** Go is authoritative. 
uCode is the operational language. TypeScript is presentation-only. 
No semantic or state logic moves into TypeScript or Python.

## Script System

**Source:** `@dev/inbox/decisions/uCODE-v2-SCRIPT_SYSTEM_SPEC.md`  
**Owner:** uDOS-core  
**Status:** locked for v2.1

### Format

Scripts are executable markdown documents with `.md` suffix (e.g., `startup-script.md`).

Structure:

```
---
id: startup
type: script
version: 2
entry: boot
permissions: [read, draw]
runtime: [tui, web]
---

# Startup

Explanation and documentation.

```ucode
SET system.status startup
DRAW BLOCK assets/banner.md
```
```

### Verb Pattern

Commands follow a strict verb syntax:

```
SET variable value        # state mutation
STATUS                    # query
WORKFLOW RUN onboarding  # execute named workflow
DRAW BLOCK file.md       # render markdown
DRAW PAT TEXT "label"    # render inline text
```

No language-style variable assignment. No implicit side effects.

## Ubuntu Workstation & Distribution

**Source:** `@dev/inbox/decisions/uDOS-v2-ubuntu-decision-detailed.md`  
**Owner:** uDOS-host  
**Status:** locked for v2.1

### Principles

1. **Minimal, Stable, Reproducible** — Ubuntu 22.04 LTS base, no bloat, deterministic build
2. **uDOS is Layered, Not Embedded** — uDOS installs on top; OS remains replaceable
3. **Sonic-Screwdriver First-Class** — Primary install via USB boot device
4. **Privacy by Default** — Proton suite for mail, docs, optional VPN
5. **Distinct Visual Identity** — OS-level uDOS branding (System 7 + modern polish)

### Repo Responsibilities

**Owns:**
- Ubuntu base image definition and package selection
- Proton integration layer
- Bootloader branding and uDOS bootstrap
- Desktop environment and theming
- Compatibility layer for uDOS-core install
- sonic-screwdriver install hooks

**Does Not Own:**
- uDOS-core runtime semantics
- uDOS-shell or extension logic
- Wizard OK control plane
- Application layers

## Wizard OK Control Plane

**Source:** `uDOS-wizard/@dev/udos-wizard-v2.1/wizard/ok/WIZARD-OK-SPEC-v2.1.md`  
**Owner:** uDOS-wizard  
**Status:** locked for v2.1

### Authority Split

- **Core** owns deterministic execution truth
- **Wizard** owns managed OK orchestration (routing, scheduling, budget, policy)
- **Dev** owns experimental harnesses and promotion readiness

Wizard remains subordinate to Core validation at all times.

### Boundary Rule

**If the capability is:**
- local, deterministic, bounded, and offline-safe → **Core**
- networked, scheduled, shared, budgeted, or deferred → **Wizard**

### OK Control Plane Functions

Wizard is not an assistant personality. Wizard is:

- a provider router
- a scheduler
- a budget enforcer
- a policy engine
- a registry host
- an audit boundary

**Managed HTTP surfaces:**
- `GET /ok/providers`
- `POST /ok/route` (with policy, budget, eligibility checks)
- `GET /mcp/tools`

## ThinUI Runtime & Theme Rendering

**Owner:** uDOS-thinui, uDOS-themes  
**Status:** scaffold ready for v2.1 Round B wiring

### State Contract

**What ThinUI is:**
- A deterministic rendering surface for boot, recovery, and system UI
- Stateless reactive renderer: state → view → frame

**What ThinUI is not:**
- A semantic system (state logic stays in Core)
- An appearance system (appearance is Themes)
- A generalized UI toolkit (specialized for uDOS system surfaces)

### State Packet (Canonical)

```typescript
export type ThinUiStatePacket = {
  view: string;                    // "boot", "recovery", "config", etc.
  mode: ThinUiMode;                // "windowed" | "fullscreen" | "recovery"
  themeId: string;                 // e.g., "thinui-c64", "thinui-modern"
  loaderId?: string;               // e.g., "sonic-screwdriver", "alpine-live"
  title?: string;
  subtitle?: string;
  status?: ThinUiStatus;           // "idle" | "running" | "warning" | "error" | "complete"
  progress?: { current?: number; total?: number; label?: string };
  actions: ThinUiAction[];
  panels?: ThinUiPanel[];
  diagnostics?: {
    safeMode?: boolean;
    offline?: boolean;
    lowResource?: boolean;
  };
};
```

### Rendering Pipeline

1. **State hydration** — Core provides state packet
2. **Theme resolution** — Resolve theme by `themeId`
3. **View rendering** — Call Themes adapter `renderThinUiState(statePacket)`
4. **Frame emit** — Publish rendered frame

### Boundary Preservation

- **Core** provides state and commands
- **ThinUI** wires state → rendering pipeline
- **Themes** owns appearance, fonts, colors, layout tokens
- Result is published (no reverse mutation)
