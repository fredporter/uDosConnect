# v2.3 Unified Spec

Status: active
Updated: 2026-03-21

## Purpose

`v2.3` is the archive-migration and productization lane after the first working
`v2.2` runtime, managed MCP, and ThinUI release baseline. The goal is to
replace archive-era dependency on older Wizard, workstation, scheduling, and
deployment surfaces with active v2 family ownership.

This spec is the umbrella reference for:

- `v2.3-rounds.md`
- `v2.3-feature-and-migration-audit.md`

## Locked Requirements

The following requirements apply across every `v2.3` round:

- strict three-lane split: Core, Wizard, and `@dev`
- canonical operating language: `@workspace`, `#binder`, and `@dev`
- binder lifecycle expressed as `compile -> package -> render -> deploy -> operate`
- Wizard remains the canonical control plane for publishing, OK agents, API budgeting, and managed external integration
- dual UI model remains required: ThinUI plus browser workflow UI
- theme output stays token-based with adapter-backed multi-target rendering
- module-system enforcement replaces legacy extension-language drift
- vault remains source of truth even when services cache or project
- education, publishing, and release surfaces remain version requirements

## Migration Priorities

### 1. Wizard recovery

Recover Wizard GUI depth from archive-era admin and browser template lineages
without importing them wholesale. The recovered operator surface must make MCP,
OK routing, assistant state, task state, and recent tool activity visible and
locally testable.

### 2. Ubuntu workstation parity

Align the Ubuntu/browser workstation lane to the browser-first local
workstation role without importing private Apple-native logic into the public
family.

### 3. Workflow-backed `@dev`

Strengthen binder, schedule, automation, and submission control so archive-era
scheduler expectations move into explicit workflow-backed v2 operations.

### 4. Sonic productization

Advance Sonic live/install/recovery delivery far enough to replace archive-era
placeholder seams with Ubuntu and Ventoy.

### 5. Archive-removal gate

Run the family validation matrix, record archive retire/freeze decisions, and
prepare the `v2.3` release evidence package.

## Release Gate

`v2.3` is complete only when:

- Wizard GUI recovery is complete and OK assistant handling is stable
- Ubuntu has a browser-workstation scaffold aligned to the browser-first local
  workstation role
- `@dev` schedules, binders, and automation control are stronger and more explicit
- Sonic live/install/recovery flows are productized enough to replace archive dependency
- archive-removal decisions are evidence-backed and rollback-safe
- the full release matrix validates cleanly across the active family
