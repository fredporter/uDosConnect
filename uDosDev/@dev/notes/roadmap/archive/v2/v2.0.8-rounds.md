# v2.0.8 Rounds

Status: active
Updated: 2026-03-21

## Purpose

Harden the dev tooling and runtime resilience layer before v2.1 convergence
work begins. This version addresses two tightly linked concerns that must be
in place before the family scales to full-operation scheduling, `@dev`
automation, and multi-repo convergence work:

1. **Dev tooling robustness** — structured logging for `@dev` scheduling and
   workflow execution, loop guards and kill-all controls for automation,
   and session-level recovery so a crashed dev run does not lose binder
   state or corrupt active surfaces.

2. **Vault survival and runtime independence** — uDOS is a snap-off OS. In the
   event of a crash or irrecoverable failure, the vault must survive. Vaults
   can be restored from `.compost` artifacts and progressive file versions.
   Sandbox protocols protect work-in-progress through draft and backup/restore
   sequences. uDOS can be broken off, reinstalled anywhere, and reconnected
   without data loss.

## Round B: Vault Survival, Snap-Off, And Sandbox Protocols

Status: completed
Owner: uDOS-core (with uDOS-dev governance)
Binder: #binder/core-v2-0-8-vault-survival

Focus:

- **Vault crash survival contract**: the vault (user data, binder state, and
  active work surfaces) must survive a runtime crash or a forced kill at any
  stage of operation. Define the minimum vault state required to resume, and
  what is considered safely discardable.

- **`.compost` restore primitive**: define the `.compost` artifact format —
  a compact periodic snapshot of vault state that survives even a full runtime
  teardown. Vaults can be restored from the most recent `.compost` plus any
  progressive file versions written since that point.

- **Progressive file versioning protocol**: any mutation to a tracked surface
  writes a versioned delta alongside the active file so that any prior state can
  be reconstructed without a full `.compost` restore.

- **Sandbox (draft and backup/restore) protocols**: work-in-progress must have
  an explicit sandbox lane — draft-before-write, backup-before-mutate. No
  promotable surface is overwritten without a backup artifact in place first.

- **Snap-off and reinstall protocol**: uDOS can be completely broken off from
  its current host environment, reinstalled on any compatible target, and
  reconnected to the family network without data loss. Define the boundary
  between the portable vault layer and the host-specific runtime bindings.

- **Reconnect sequence**: define the canonical reconnect handshake — vault
  identity verification, family network re-registration, and active binder
  state resync.

Working tags:

- @dev/v2-0-8-vault
- @dev/vault-survival
- @dev/snap-off

Breakdown:

1. define vault survival contract in `uDOS-core/contracts/vault-survival.md`:
   minimum crash-safe state, discard-safe state, and recovery priority order
2. define `.compost` format: header (vault-id, timestamp, version-vector),
   compressed state blob, integrity checksum; document rotation policy and
   max restore depth
3. define progressive file versioning protocol: delta format, retention window,
   and resolution order when restoring from `.compost` + deltas
4. define sandbox protocol: draft-before-write guard, backup-before-mutate
   trigger conditions, and restore-from-backup sequence
5. define snap-off protocol: what is portable (vault), what is host-bound
   (runtime bindings), and the clean-detach sequence
6. define reconnect handshake: vault identity claim, family network reattach,
   binder-state resync, and conflict resolution on diverged surfaces
7. document full restore sequence: bootstrap order, minimum required state to
   resume a binder mid-round after catastrophic failure
