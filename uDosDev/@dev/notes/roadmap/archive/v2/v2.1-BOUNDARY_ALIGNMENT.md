## Semantic Boundary Lines

### Core → Everything Else (Downward Authority)

**Core publishes:**
- Contracts (state shapes, commands, vault semantics)
- Deterministic execution truth
- Script runtime semantics
- uCode verb dictionary
- Vault survival protocol

**Consumers depend on Core but never redefine it:**
- Wizard routes within Core semantics
- ThinUI renders Core state
- Themes style Core-defined state packets
- Ubuntu carries Core on top

**Violation pattern:** If you catch yourself adding semantic logic to Wizard, ThinUI, or Themes, move it to Core.

### Wizard → ThinUI / Themes (Orchestration, Not Definition)

**Wizard can:**
- Trigger ThinUI rendering by publishing state
- Request theme or loader by name
- Schedule / defer rendering operations
- Log audit trail of rendering events

**Wizard cannot:**
- Define what state means (that's Core)
- Define appearance (that's Themes)
- Mutate Core state directly (must go through Core)

### ThinUI → Themes (Rendering Pipeline, Not Backward Mutation)

**ThinUI publishes state packet to Themes:**

```
ThinUI: "Here is a state packet with themeId = 'thinui-c64'"
Themes: "I will render that packet using c64 appearance"
Result: frame emitted
```

**Themes do NOT:**
- Mutate state
- Change themeId mid-render
- Add semantic fields to the packet
- Call back into ThinUI

### Ubuntu → uDOS-Core (Carrier, Not Owner)

**Ubuntu provides:**
- Boot environment
- Package management
- System services
- Hardware abstraction

**Ubuntu does NOT:**
- Define uDOS semantics
- Own extension logic
- Manage binder lifecycle
- Route Wizard operations

## Data Flow Model

### Boot Sequence

```
Ubuntu bootloader
       ↓
    Proton
       ↓
      uDOS-core initialize
       ↓
   Wizard OK start (routes existing uCode blocks)
       ↓
  generate ThinUiStatePacket
       ↓
   ThinUI resolve(themeId)
       ↓
   Themes adapter render(statePacket)
       ↓
   Frame emitted → display
```

### State Mutation Sequence

```
User action (TUI key or click)
       ↓
   ThinUI emit event
       ↓
  Wizard route to handler
       ↓
   Core execute command (uCode verb)
       ↓
   Core publish new state
       ↓
   ThinUI re-render with new state
```

### Script Execution

```
User: "RUN startup-script.md"
       ↓
   Core parse script frontmatter + uCode blocks
       ↓
   Dispatch uCode verbs (SET, STATUS, WORKFLOW, DRAW)
       ↓
   ThinUI renders DRAW outputs
       ↓
   Themes apply style to render request
       ↓
   Frame → display
```

## Cross-Boundary Communication Contracts

### (1) Core ↔ Wizard

**Direction:** Wizard subscribes to Core  
**Protocol:** Eventbus or publish-subscribe  
**Payload:** State changes, command results, audit  

**Example:**
- Wizard: "Subscribe to system.status changes"
- Core: "Publishes {status: 'startup', timestamp: ...}"
- Wizard: "Logs audit, routes to MCP providers"

### (2) Wizard ↔ ThinUI

**Direction:** Wizard publishes, ThinUI renders  
**Protocol:** Idempotent state publishing  
**Payload:** ThinUiStatePacket + action results  

**Example:**
- Wizard: "Render view='boot', status='running', progress=50"
- ThinUI: "Received state, resolving theme 'thinui-c64', rendering..."
- Wizard: "Logs render begin + end timestamps"

### (3) ThinUI ↔ Themes

**Direction:** ThinUI requests render, Themes returns frame  
**Protocol:** Function call or service interface  
**Payload:** ThinUiStatePacket → canvas/frame  

**Example:**
```typescript
const frame = await resolveThinUiTheme(themeId).renderState(statePacket);
```

### (4) Core ↔ ThinUI (Direct)

**Direction:** One-way (for recovery/diagnostics)  
**Protocol:** Call optional safe-mode rendering  
**Payload:** Minimal state (no Themes, no Wizard)  

**Example:**
- Core (in safe mode): "Direct render status='error', mode='recovery'"
- ThinUI: "Use fallback renderer (no Themes), output text+minimal styling"
