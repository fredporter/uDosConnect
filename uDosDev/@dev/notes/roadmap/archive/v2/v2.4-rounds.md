# v2.4 Rounds

Status: completed
Updated: 2026-03-29

## Purpose

v2.4 is the next family plan after the closed `v2.3` archive-migration lane.

This round opens a workspace-led upgrade sequence that adds a new browser and
ThinUI-facing operator shell, defines a canonical markdown-first intake and
conversion lane, establishes controller input as a shared family contract, and
stages an optional Deer Flow execution backend without moving truth ownership
out of the existing Core, Wizard, and vault boundaries.

v2.4 is complete only when:

- `uDOS-workspace` is activated as a distinct family repo with binder-facing
  contracts and a prototype shell
- MDC is documented and staged as the canonical intake and normalization layer
  for file import into markdown-first vault workflows
- UCI contracts and shell-first controller surfaces are published without
  forcing controller-first behavior onto unrelated product lanes
- Deer Flow is staged as an explicitly optional execution backend with
  translation and normalized result contracts
- the family validation and promotion lane records what shipped and what
  remains intentionally deferred

  distinct repo rather than a package inside another repo
- workspace docs make the operator-of-truth versus owner-of-truth boundary
  explicit

  truth for browser workspace compilation
- the spatial/layer model is documented as a visualization consumer of Core
  spatial contracts
- the prototype scope is legible through docs, scaffolded packages, sample
  binders, and example compile manifests
- Round B binder is open with owner and acceptance criteria

Notes:

- workspace remains a consumer of Core contracts, Wizard jobs, and Empire
  panels rather than taking ownership of those domains

## Round B: MDC Intake And Conversion Contract And Engine MVP

Status: completed
Owner: uDOS-core
Binder: #binder/core-v2-4-mdc-conversion-engine

Focus:

- define MDC as the canonical intake and normalization layer for file import
  into markdown-first vault workflows
- publish supported-type matrix, deterministic conversion rules, output routing
  rules, and vault placement doctrine
- implement the MVP behavior locked by the pack: markdown generation,
  non-destructive source handling, spreadsheet JSON/SQLite thresholds, PDF
  text-only extraction, and Marp-style PPT direction where supported
- keep launch and open behavior out of Core ownership; Core owns contracts and
  normalization semantics

Exit criteria:

- MDC intake contract and supported-type matrix are published in Core-owned
  docs or contracts
- normalized output expectations are explicit for `.md`, `.json`, and
  `.sqlite`
- vault routing rules for converted artifacts are documented
- fixture-based conversion checks are defined for representative file classes
- Round C binder is open with owner and acceptance criteria

Notes:

- v2.4 only requires contract-plus-MVP conversion behavior, not a full
  rich-media ingestion platform
- original files must remain untouched; only normalized derivatives are
  persisted into the vault

## Round C: UCI Shared Controller Input And Shell-First UX Prototype

Status: completed
Owner: family
Binder: #binder/family-v2-4-uci-controller-surfaces

Focus:

- publish the canonical UCI event model, controller mapping schema, prediction
  provider contract, mode model, and reserved semantic actions
- land a shell-first prototype for command palette and radial keyboard UX
  without forcing controller-first behavior onto all surfaces
- define handoff expectations for `uHOME` and Gameplay while keeping first
  implementation depth in Core plus Shell
- keep remote ranking optional and non-blocking while local prediction layers
  remain the default path

Exit criteria:

- controller mapping JSON schema and prediction schema are published
- the shared mode set is explicit: `nav`, `text`, `command`, `edit`, `game`,
  `media`
- reserved semantic actions are documented and stable across Core and Shell
- shell-first prototype direction is documented for command palette and radial
  keyboard behavior
- Round D binder is open with owner and acceptance criteria

Notes:

- UCI is universal input infrastructure, not a controller-only fork of uDOS
- broad rollout to every product repo is deferred until after the first
  prototype round

## Round D: Deer Flow Optional Execution Adapter And Wizard Backend Lane

Status: completed
Owner: uDOS-wizard
Binder: #binder/wizard-v2-4-deerflow-execution-adapter

Focus:

- activate `uDOS-plugin-deerflow` as an optional execution-backend repo or
  tracked plugin lane
- implement only the MVP portions needed for first roadmap inclusion:
  translation MVP plus controlled execution MVP
- keep binder semantics, compile truth, vault persistence, publish routing,
  trust classes, and certification boundaries in uDOS
- expose Deer Flow as an explicitly optional Wizard-selectable backend for
  long-horizon or graph-heavy workflows

Exit criteria:

- the workflow-to-Deer Flow translation schema is published
- the normalized execution-result schema is published
- Wizard backend-selection doctrine is explicit for native versus Deer Flow
  execution lanes
- trust and policy boundaries for optional graph execution are documented
- deferred items are explicit: certified workflow set, remote clusters, graph
  editing, and memory sync import/export
- Round E binder is open with owner and acceptance criteria

Notes:

- Deer Flow graphs are generated execution artifacts, not source-of-truth
  workflow state
- v2.4 keeps the adapter experimental and optional rather than release-gating
  the family on certification depth
