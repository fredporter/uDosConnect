## What Is Clearly Landed In v2

### uCODE runtime baseline

Landed:

- Core owns the runtime and command authority
- `RUN <script.md>` and `SCRIPT RUN <path>` are implemented
- uppercase uCODE verb parsing and minimal dispatch are implemented
- script document and execution contracts are published

Evidence:

- `uDOS-core/contracts/ucode/ucode-verb-contract.json`
- `uDOS-core/contracts/script-document-contract.json`
- `uDOS-core/contracts/ucode-script-execution-contract.json`
- `uDOS-core/udos_core/ucode.py`
- `uDOS-core/udos_core/runtime.py`
- `uDOS-core/udos_core/script_runtime.py`

Assessment:

- language/runtime spec is partially realized in a strong baseline form
- the deterministic DSL model is present
- the full workflow/binder/scheduler engine implied by the spec is not yet fully realized

### Wizard MCP and Shell managed consumption

Landed:

- Wizard owns the MCP bridge
- VS Code-facing MCP stub exists
- Shell can initialize, list, and call MCP tools
- `ok.route` exists as a managed route surface

Evidence:

- `uDOS-wizard/mcp/vscode-extension/`
- `uDOS-wizard/wizard/main.py`
- `uDOS-shell/internal/localexec/wizard.go`
- `uDOS-shell/internal/app/model.go`

Assessment:

- v2 has the first real managed MCP/operator bridge
- this satisfies the v2.2 baseline but not the richer operator GUI and assistant handling target

### ThinUI and launcher line

Landed:

- real first-frame output exists
- `thinui-c64`, `thinui-nes-sonic`, and `thinui-teletext` exist
- Alpine, Ubuntu, and Sonic ship executable ThinUI demo/launcher handoffs
- Shell exposes startup health and demo entrypoints

Evidence:

- `uDOS-thinui/src/runtime/bootstrap.ts`
- `uDOS-thinui/src/views/teletext-display.ts`
- `uDOS-themes/src/adapters/thinui/`
- `uDOS-alpine/docs/thin-gui-launcher-integration.md`
- `uDOS-host/docs/first-run-story.md`
- `sonic-screwdriver/scripts/demo-thinui-launch.sh`
- `uDOS-shell/internal/localexec/family.go`

Assessment:

- the ThinUI/launcher line has moved from scaffold to visible, testable output
- it is still a first-render lane, not a full application surface

### Sonic / Ventoy / Ubuntu baseline split

Landed:

- repo ownership split is now clear between `uDOS-host`, `sonic-ventoy`, and `sonic-screwdriver`
- Ventoy-compatible templates and GPT layouts exist
- Sonic can initialize, add, update, and theme sonic-stick surfaces
- Ubuntu exposes install hooks and bootstrap examples

Evidence:

- `sonic-ventoy/config/partition-layouts/`
- `sonic-ventoy/templates/ventoy/`
- `sonic-screwdriver/apps/sonic-cli/cli.py`
- `sonic-screwdriver/services/sonic_stick_integration.py`
- `uDOS-host/sonic-hooks/`

Assessment:

- the v2 packaging/deployment boundary is much healthier than v1
- the policy and starter surfaces exist
- full end-to-end image composition and install fidelity are still incomplete

## What Is Only Partially Migrated

### Script system beyond raw uCODE blocks

The script spec promises:

- story blocks
- presentation blocks
- TypeScript-rendered UI workflows
- portable living documents

Current state:

- Core loads markdown, frontmatter, and uCODE blocks
- contracts describe `story` and web/TypeScript handling
- no equivalent full story runtime or interactive document engine is yet obvious in the active repos

Assessment:

- script execution exists
- script-driven interactive UI/document behavior is still partial

### Wizard GUI

Current state:

- active Wizard has browser/operator surfaces and automation panels
- MCP integration is real
- `docs/gui-assessment-and-plan.md` explicitly states the current repo is still not a fully recovered operator GUI product
- archived `uDOS-v1-8-archived/apps/admin` and `wizard/web/templates` contain richer workflow/UI source material

Assessment:

- GUI direction exists
- GUI detail and stabilization are correctly staged as `v2.3` Round A

### OK assistant handling

Current state:

- `ok.route` exists
- Wizard docs define provider and MCP architecture
- Shell can call managed MCP routes

Missing compared to desired v2 posture:

- explicit assistant modes across surfaces
- clearer retry/failure/defer behavior
- stronger operator-facing state for current assistant work

Assessment:

- first route exists
- full assistant handling is not complete yet

### Workflow-driven `@dev` scheduling and binder operation

Current state:

- `uDOS-dev` has binder workflow, reports, scheduler audit logs, and operations checks
- Shell exposes workflow and automation contract inspection plus queue/result examples
- archive-era systems carried stronger schedule and background-job expectations

Missing:

- one explicit workflow-backed operating model for binders + schedules + submissions
- clearer bridge between scheduled work and binder lifecycle
- durable and visible scheduled dev execution lane

Assessment:

- governance tooling exists
- the workflow-driven operating model is still incomplete

## Major Archive-Derived Gaps Still Open

### Ubuntu browser workstation migration and OBSC lineage

The Ubuntu decision document is ambitious. It promises the Ubuntu/browser
implementation of the local binder workstation with Proton-centered convenience
surfaces and browser GUI operation.

Archive source that matters:

- `OBSC-mac-app-v1`

Key features in archive not yet visibly migrated into active v2:

- calendar / reminders / notes / contacts / mail import workflows
- local workstation-grade editor and panel workflows
- task scheduler / background automation control surface
- binder-native workstation views with personal data pipelines

Assessment:

- v2 has the platform split and the browser-workstation decision
- Ubuntu should now target the browser-first local workstation role while also
  covering the binder-workstation capabilities that still only exist in the
  archive lineage

### Sonic deployment ambition versus current delivery

The Sonic brief promises:

- Ventoy-class multi-image boot
- uDOS-styled sonic menu
- config injection + identity
- live / install / recovery parity
- portable OS orchestration, not just a flasher

Current v2 state:

- strong planning and starter integration are present
- templates, manifests, GPT layouts, and dry-run/apply surfaces exist
- Ubuntu hook integration exists but remains partly placeholder-backed

Assessment:

- the architecture is in the right repo split now
- the full product ambition from the brief is not fully delivered yet

### Archived Wizard / admin GUI recovery

Archive source that matters:

- `uDOS-v1-8-archived/apps/admin`
- `uDOS-v1-8-archived/wizard/web/templates`
- `uDOS-v1-8-archived/modules/thin-gui`

Current v2 state:

- current Wizard owns the correct boundary
- current GUI remains lighter than the archived operator surfaces
- `uDOS-wizard/docs/gui-assessment-and-plan.md` already identifies this recovery gap

Assessment:

- this is an explicit v2.3 migration target, not closed work

### Archive-era scheduling and automation expectations

Archive sources that matter:

- `uHOME-server-v1-archived/services/scheduling/README.md`
- `OBSC-mac-app-v1/docs/user-guide/automation.md`
- `OBSC-android-app-v1-archived/06-ROADMAP-MILESTONES.md`

Current v2 state:

- contracts and sample flows exist
- durable background scheduling and operator-grade schedule control are not yet unified in the active family

Assessment:

- this gap should remain attached to `v2.3` Round C

## Recommended v2.3 Handling

### Round A

Recover and stabilize Wizard GUI detail using:

- `uDOS-wizard/docs/gui-assessment-and-plan.md`
- archived `uDOS-v1-8-archived/apps/admin`
- archived `uDOS-v1-8-archived/wizard/web/templates`

### Round B

Treat `ok.route` as the starting point, not the finish line:

- explicit assistant state model
- clearer defer / retry / failure paths
- Shell and Wizard parity for assistant semantics

### Round C

Close the scheduler/binder/workflow gap:

- workflow-backed binder progression
- schedule-backed `@dev` routines
- documented human-review boundaries

### Round D

Use family integration to close the biggest archive migration gaps that now have
active repo homes:

- Wizard operator GUI lineage
- Ubuntu browser-workstation lineage
- Sonic install/live/recovery lineage
- script/story and first-run operator flows

## Bottom Line

The v2 family is no longer missing its foundation. The runtime split, MCP
ownership, ThinUI launcher line, and deployment boundaries are real.

The main unfinished work is not foundational rebuild anymore. It is migration
and stabilization work:

- recover richer Wizard GUI behavior from archived lineages
- close assistant handling gaps
- turn `@dev` binder and schedule flow into a real workflow-driven operating model
- migrate the workstation-grade and scheduler-grade features that still live only
  in archive-era products and notes
