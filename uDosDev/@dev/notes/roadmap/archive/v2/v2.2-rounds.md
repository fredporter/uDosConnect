# v2.2 Rounds

Status: completed
Updated: 2026-03-21

## Purpose

v2.2 is the implementation lane for canonical decisions locked in v2.1.

v2.1 locked:
- uCODE language/runtime spec (owner: uDOS-core)
- uCODE script system spec (owner: uDOS-core)
- Ubuntu browser-workstation strategy (owner: uDOS-host)
- Wizard OK control-plane boundary (owner: uDOS-wizard)
- ThinUI state contract and runtime scaffold (owner: uDOS-thinui)
- Networking boundaries (since v2.0.4, owner: uDOS-wizard)

v2.2 builds on those decisions. It is complete only when:
- Core ships a working uCODE verb runtime and script dispatch
- Wizard ships a working MCP ↔ VS Code integration surface
- ThinUI produces a first real visible rendered output via theme adapter
- The full family remains validated and promotion-ready

---

## Round A: uCODE Runtime and Script Dispatch

Status: completed
Owner: uDOS-core
Binder: #binder/core-v2-2-ucode-runtime

Focus:

- implement uCode verb parser and dispatch in Go runtime (Core)
- implement uCode script execution engine (`.md` script format with frontmatter)
- publish Core script execution contract for other repos to consume
- define test coverage for basic verb set: SET, STATUS, WORKFLOW RUN, DRAW BLOCK, DRAW PAT TEXT

Exit criteria:

- `uDOS-core` ships a `ucode/` runtime module with parser, verb dispatch, and environment
- script execution contract published in `contracts/`
- unit tests cover happy-path and error-path for each verb
- Shell can invoke a `.md` script via Core
- Round B binder is open with owner and acceptance criteria

Prereqs:

- v2.1 promoted (spec locked and in main)
- v2.1-CANONICAL_DECISIONS.md is the canonical authority

Notes:

- No UI, no Wizard interaction in Round A — this is Core-only runtime work
- Keep verb set minimal: only implement verbs with defined spec entries
- Script execution contract must not embed networking — keep it offline-safe

## Round C: ThinUI First Real Render and Themes Output

Status: completed
Owner: uDOS-thinui
Binder: #binder/thinui-v2-2-first-render

Focus:

- advance ThinUI from runtime scaffold to first real visible output
- wire `renderThinUiState` → `resolveThinUiTheme` → `thinui-c64` adapter → frame emission
- produce a real rendered frame: boot screen, title, status, and at least one action
- validate live render path on at least one target runtime (Alpine or sonic kiosk boot)
- document: frame output format, how to add a new theme adapter

Exit criteria:

- `thinui-c64` adapter produces a real rendered frame (not a placeholder)
- boot screen is visually identifiable and contains expected elements (banner, status, action)
- Alpine Thin GUI launcher invokes ThinUI and terminates cleanly
- frame output format documented in `contracts/frame.ts` or equivalent
- Round D binder is open with owner and acceptance criteria

Prereqs:

- v2.1 promoted (runtime scaffold in main)
- uDOS-themes `resolveThinUiTheme` and adapter loader in place (from v2.1)
- Alpine launcher contract in place (from v2.1)

Notes:

- Round C owns visible output, not Core logic
- ThinUI consumes Core state but does not define it
- Frame format should be stable enough for a second adapter to consume
- Do not add state semantics to ThinUI — rendering only
