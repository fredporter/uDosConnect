# v2.0.5 Shared Ingest And Workflow Kernel

Status: active working direction  
Updated: 2026-03-15

## Purpose

Define the smallest shared processing kernel that public `uDOS` repos and the
private `omd-mac-osx-app` can implement in parallel without becoming runtime
dependencies of each other.

## Core Rule

Share contracts, vocabulary, fixtures, and deterministic processing stages.

Do not share:

- Apple-specific adapters
- provider-specific adapters
- UI state or application shells
- repo-local SQLite layouts
- network transport implementations

This keeps public and private development aligned while preserving separate
runtime ownership.

## Shared Objective

Both streams are trying to do the same high-level work:

1. ingest external or operator-provided material
2. classify it into meaningful workflow content
3. extract structure and metadata
4. normalize it into canonical records
5. progress deterministic workflows
6. export or hand off selected results downstream

The shared layer should therefore live at the processing-kernel level, not at
the adapter or app-runtime level.

## Shared Kernel Surface

`uDOS-core` should own the public contract for these stages:

1. `Acquire`
2. `Classify`
3. `Extract`
4. `Normalize`
5. `Store`
6. `Progress`
7. `Export`

These are semantic stages, not a mandate that every implementation use the same
programming language or persistence layout.

## Recommended Shared Types

These are the main shapes worth promoting into shared contracts or fixture
definitions.

### Input and processing envelopes

- `AcquireInput`
- `ImportedPayload`
- `ClassifiedPayload`
- `ExtractedPayload`
- `NormalizedRecordBundle`
- `StoredRecordBundle`

### Workflow progression

- `WorkflowIntent`

- `WorkflowProgressEvent`
- `WorkflowOutcome`
- `JobOutcome`

### Export and handoff

- `ExportEnvelope`
- `ProjectionEnvelope`

### Shared vocabulary

- record types
- source-system ids
- workflow states
- sync states
- conflict states
- compile and render status vocabulary

## Ownership Split

### `uDOS-core` owns

- stage semantics
- canonical record and envelope vocabulary
- shared schemas
- sample payloads and fixture sets
- deterministic progression rules
- compile and export semantics that are family-wide

### `uDOS-wizard` owns

- online orchestration and dispatch
- provider and MCP bridges
- preview and publish transport
- callback and result routing

### `uDOS-empire` owns

- always-on remote sync execution
- CRM and provider-specific mapping lanes
- queueable sync-package execution and handoff
- webhook and cron-style remote workflow surfaces

### `omd-mac-osx-app` owns

- Apple Mail, Apple Contacts, AppleScript, and native macOS adapters
- local desktop review flows
- app-local contact databases
- private workflow intelligence that is not yet family-general
- app-local runtime index and operational UX

## Parallel Development Rule

The public and private streams may implement the same kernel independently.

When one stream advances first:

- the new behavior should be expressed in shared contract language if it is
  truly reusable
- the other stream should be able to slot that behavior in later without
  architecture rewrites
- no repo should require importing implementation code from the other repo

This means the synchronization mechanism is:

- shared contract docs
- machine-readable schemas where appropriate
- golden fixtures and example envelopes
- mirrored tests against the same fixtures

not:

- direct source dependency
- copy-paste of adapters
- shared app/runtime layers

## Recommended Development Pattern

Use this order when new ingest or workflow power appears:

1. Prove it in the owning repo.
2. Decide whether the behavior is app-specific, provider-specific, or
   family-general.
3. If family-general, promote the contract and fixture into `uDOS-core`.
4. Keep each repo's implementation local.
5. Add fixture-based conformance tests in every consumer repo.

## Promotion Criteria

Promote a behavior into `uDOS-core` only if it is:

- deterministic
- platform-neutral
- not provider-specific
- not Apple-framework-specific
- useful across more than one repo
- explainable as a stable public contract

Examples that likely qualify:

- normalized message record shape
- contact evidence vocabulary
- workflow progression status model
- compile/export envelope shapes
- conflict and sync-state vocabulary

Examples that likely do not qualify:

- Apple Mail acquisition code
- HubSpot field mapping details
- app-specific SQLite migrations
- Wizard callback store implementation
- UI review workflows

## Initial Concrete Action

Near-term shared-kernel work in `uDOS-core` should focus on:

- extending existing sync-record and runtime-service contracts rather than
  inventing a second processing model
- adding ingest/workflow fixture docs and examples
- documenting the common pipeline stages in canonical public language

## Bottom Line

Yes, both streams should share a version of the same ingest and workflow
implementation model.

They should share the kernel contract, not the adapters or app runtimes.
