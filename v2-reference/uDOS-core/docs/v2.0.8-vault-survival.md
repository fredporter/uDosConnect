## Overview

v2.0.8 Round B publishes the Core vault survival contract. The vault is the
portable data layer that must survive crashes, forced kills, snap-off events,
and host reinstalls without data loss.

This round defines five interconnected protocols:

1. **Crash-safe/discard-safe state boundary** — what must survive and what can be regenerated
2. **`.compost` snapshot format** — compact periodic vault snapshot with integrity checksum
3. **Progressive file versioning** — write-ahead delta log for point-in-time restore
4. **Sandbox protocol** — draft-before-write and backup-before-mutate guards
5. **Snap-off and reconnect** — portable vs host-bound boundary, detach sequence, and reconnect handshake

## Key Definitions

### Vault Layer

The vault is everything that must be portable and crash-safe:

- `.compost/` snapshots and deltas
- `@dev/logs/checkpoints/`  
- `@dev/logs/workflow-scheduler-audit.jsonl`
- Active `@dev/` binder surfaces

### `.compost` File

Three-section JSONL format: header → state surfaces → footer (with checksum).
Named `<vault-id>-<timestamp>.compost`. Rotation: keep last 3.

### Delta Record

Written before every mutation to a tracked surface. Enables point-in-time
restore between snapshots. Retain minimum 10 deltas per surface.

### Sandbox Guards

- **Draft-before-write**: write to `.draft`, verify, then rename atomically.
- **Backup-before-mutate**: copy to `.bak-<timestamp>` before any mutation; remove on success, restore on failure.

### Snap-Off Boundary

| Portable (vault) | Host-bound (not portable) |
| --- | --- |
| `.compost/` | `.venv/` |
| `@dev/logs/checkpoints/` | `node_modules/` |
| `@dev/` surfaces | PID files |
| Binder state files | Build artifacts |

## Boundary

- `uDOS-core` owns these protocol definitions. No other repo may redefine them.
- `uDOS-grid` datasets are not included in vault snapshots.
- `uDOS-dev` governs which `@dev/` surfaces are tracked in the vault.
- Runtime bindings (`.venv`, build outputs) are host-owned; Core does not define their recovery.
