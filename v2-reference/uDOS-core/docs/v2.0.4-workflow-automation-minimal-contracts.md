# v2.0.4 Workflow and Automation Minimal Contracts

## Purpose

Lock the minimal shared contract layer between:

- `uDOS-wizard` as workflow authority
- `uHOME-server` as always-on automation substrate
- app surfaces as user-touch advancement surfaces
- `uDOS-gameplay` as interpretation and progression consumer

This note does not assign implementation ownership to `uDOS-core` beyond the
shared contract layer.

## Runtime Split

- `uDOS-wizard` owns workflow state transitions that require conscious user
  intent, approval, or online assist
- `uHOME-server` owns durable execution of scheduled and queued local work
- app surfaces may expose approve, resume, or advance actions without becoming
  the workflow authority
- `uDOS-gameplay` may react to workflow and job events without becoming the job
  runner

## Minimal Shared Contracts

### `workflow-state`

Artifact:

- `contracts/workflow-state-contract.json`

Required fields:

- `workflow_id`
- `step_id`
- `status`
- `awaiting_user_action`
- `last_transition_at`
- `origin_surface`

Purpose:

- describe where a workflow currently is
- tell local runtimes whether work is blocked on user action
- let Wizard, app surfaces, and gameplay interpret the same state

### `workflow-action`

Artifact:

- `contracts/workflow-action-contract.json`

Required fields:

- `workflow_id`
- `action`
- `requested_by`
- `requested_at`
- `policy_flags`

Purpose:

- represent explicit user-meaningful transitions such as advance, pause,
  approve, reject, or request-assist
- keep conscious control events distinct from background execution

### `automation-job`

Artifact:

- `contracts/automation-job-contract.json`

Required fields:

- `job_id`
- `requested_capability`
- `payload_ref`
- `origin_surface`
- `policy_flags`
- `queued_at`

Purpose:

- let Wizard or another client ask the always-on runtime to fulfill work
- keep the execution request stable even when the initiating surface closes

### `automation-result`

Artifact:

- `contracts/automation-result-contract.json`

Required fields:

- `job_id`
- `status`
- `output_refs`
- `event_refs`
- `completed_at`
- `suggested_workflow_action`

Purpose:

- report what the automation substrate completed
- give Wizard or app surfaces enough information to continue the workflow

### Supporting Shared Primitives

- `artifact-ref`
- `capability-registry`
- `event-log`

These remain small and reusable. They should not absorb repo-specific policy.

## Boundary Rules

- `uDOS-core` owns the shape of these shared contracts
- `uDOS-wizard` owns workflow policy and approval semantics
- `uHOME-server` owns automation fulfillment semantics
- app repos consume the contracts without becoming the source of truth
- `uDOS-gameplay` may read and interpret these contracts for `LENS`, `SKIN`,
  and progression feedback

## Product Rule

Wizard does not need to stay open for routine local fulfillment.

Wizard may still be required for:

- online API assist
- mission replanning
- explicit approvals
- user-conscious project steering

This is a product boundary, not a failure mode.
